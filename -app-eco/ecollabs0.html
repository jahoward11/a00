Import or Create a Project Database
  1. Tap All DBs (top right corner of screen, or last item of the Navbar ≡ menu) to return to the web app entryway.
  2. In the DB‑open input field, enter a name for a new (or, pre-existing) project database.
  3. Tap OPEN (while online).
  4. If prompted, log in.
Note: Import/creation of a database may take a minute or two to complete. However, nothing will happen unless the user has been granted access by the team administrator.
Preview and Publish a Document
  1. Before or after making (final) changes to a publmgr build file, give it a filename in the File-load input field of the Craft control panel—then tap SAVE, to temporarily preserve the changes locally (i.e., on the device only).
    Optional: When naming the file, prepend the filename with a subdirectory name (and slash) to organize it together with other, related project files—e.g., "mysubdir/myfilename".
  2. Tap the Preview tab to render and display the resulting web doc that is generated from the contents of the publmgr file—for final review and approval.
  3. When ready to publish, navigate to Final Prep & Publish > Prepress.
  4. Select a local and corresponding, remote database—then tap Import Data.
    Tip: This step ensures that you have all the current file changes made by others before you submit your file changes.
    Alert: If two or more people are editing the same file at the same time, the more recently committed version will overwrite the changes made in a previously committed version—thus, someone's changes will be lost.
  5. Navigate to Final Prep & Publish > Commits.
  6. Follow each of the four numbered steps, in sequence—
    (1) to formally insert a publmgr build file into the project database;
    (2) to generate and insert a resulting web doc;
    (3) to share the publishing files (build files and web doc) with the team; and, finally,
    (4) to publish the web doc to a public server.
  7. After confirming that a working file has been committed (updated) to the database, clean up unecessary files on the device by deleting the original, temporary file—found under LOCAL temporary files in the File-load select list; a file now flagged with an asterisk and revision number.
Note: The Prepress step and each of the four Commit steps may also be executed individually and out of sequence—for various purposes.
Communicate with Team Members
  1. Log in from the web app entryway.
  2. Load a publishing file by selecting one from either the Preview select list (top of screen, or first item of the Navbar ≡ menu) or the File-load select list (under the Craft tab of the Navbar).
  3. Tap the Discuss tab (on the Navbar, or in the Navbar ≡ menu) to display the Project Discussion screen.
  4. Scroll to the end—then fill out the Message field and optional Subject field with a memo related to the loaded publication.
  5. Tap ADD to save the communication locally (i.e., on the device only).
  6. Tap REFRESH to then send (and receive) all newly shared communications to/from all team members.
Access the HTTP Console
Function overview: Unlike most editing apps, this app gives the user full control over the management of their data. Full control means control over both their local (device) and their remote (server) databases.
 In most cases, a local database is a cloned copy of its counterpart, remote database—and vice versa. Add in a second team member with their own device and we create a third, cloned copy. Maintaining two (or more) identical versions of data is useful for many reasons. But, doing so introduces sophisticated processing tasks. The most common of these tasks are built into the app. But, many more possibilities exist to manage data in special, even unique, ways.
 Learning to use the HTTP console is analogous to learning how to run computer operations through the command line interface (CLI). After getting the basics of HTTP (and CouchDB) down, a user could potentially use just their handheld device to wield absolute, real-time control over all of their remote DB servers—both private and public.
Alert: Along with greater power comes greater hazards. Because a user of this app is enabled with greater control over their data, basic precautions should be followed—such as, "sync all data before beginning work," "save your work as you go," "test to confirm transmission was received," etc.
  1. Tap Craft (or, the Navbar ≡ menu symbol, if present) to open the Craft control panel of the Navbar (at top of screen).
  2. Toggle the HTTP console open and closed by tapping the small, blue triangle in front of the console label—a label that looks like this:
    ▶ HTTP: Database Query Console
  3. Technical use instructions are accessible within the HTTP console. When the console is blank, tap (once or twice) the SyncD, the RetrvD and/or the SubmD button.
View Differences Between Two Files
  1. Load a project file from the File-load select list of the Craft control panel of the Navbar (at top of screen).
  2. Check the nearby, Pty Diffs checkbox.
  3. Load a second, similar (e.g., an earlier version) project file from the 2nd-file select list.
  4. In the JSO-Pty Edit screen, a green outline around a property field indicates corresponding properties with identical data; a yellow or red outline indicates corresponding properties with data differences.
  5. DIFF: If a more detailed view of the text differences of a single property is desired, generate a "diff" file by taking the following steps:
    1. Save the working file (locally or remotely) if any changes have been made to it.
      Note: Loading a "diff" file (as directed in subsequent steps) replaces both of the preloaded files that are being compared—and, thus, is prevented if the primary, working file has unsaved changes.
    2. Locate the paired fields of the property being analyzed—in the JSO-Pty Edit screen.
    3. Focus (activate) the upper field—then tap the diff (⬓) symbol that appears adjacent to the field.
    4. The two versions of the common property are automatically displayed together in the Preview screen—stylized with color coding to show word- and line-level changes. Deleted lines of text are colored solid red (in the original text field), inserted lines of text are colored solid green (in the edited text field), and altered words within a line are colored yellow (in both text fields).

Choose Login or No-Login App Re-entry
Login-entry option: A fresh login (i.e., after 60 minutes since last login) is required only in these four scenarios:
- For a first-time setup, as when joining a new team;
- For resuming collaborative work after having manually logged out;
- For cloning a remote, project database, as when a new project has just been introduced by a team leader;
- For performing an administrative task, such as changing a team-member's role or creating a new project database.
  1. Navigate to Custom Tools > App.
  2. In the Start-up-default-database select list (first one), select the blank option (at top of list).
  3. Tap All DBs (top right corner of screen, or last item of the Navbar ≡ menu) to return to the web app entryway.
    Alternatively, close and re-launch the app.
  4. Clear the DB‑open input field (if necessary).
  5. Tap OPEN (while online).
  6. As prompted, log in.
No-login-entry option: Return anytime without logging in again. A no-login re-entry should become the usual practice.
Why? Taking extra time to log in fresh for every session is generally unnecessary—as when simply resuming work in a previously imported, project database.
How? Configure the app (as follows) to bypass the login screen when re-entering the app.
  1. Navigate to Custom Tools > App.
  2. In the Start-up-default-database select list (first one), select any but the blank option.
  3. Tap All DBs (top right corner of screen, or last item of the Navbar ≡ menu) to return to the web app entryway.
    Alternatively, close and re-launch the app.
  4. The DB‑open input field now shows either the current or the default database name.
    Optional: Use the DB‑open select list to select the blank option (which will close an open DB) or else, any other, listed database name (which will open a different DB).
  5. Tap OPEN.
No-login-entry option while logged out: Entering the app while logged out (or, without ever having logged in) is possible—though unusual. Doing so disables online-related functions and accesses a limited set of offline, non-collaborating functions—as when wanting just to practice coding or to test-drive the app before joining a team.
Alert: When logged out, do not attempt to do any team-collaboration work. An unidentified user could not contribute interactively—e.g., post comments, view real-time changes being made by others, submit fresh document revisions, etc.
  1. Log out (if necessary).
  2. Close the app.
  3. Set device to airplane mode to sever any internet connection.
  4. Re-launch the app.
  5. Tap OPEN.

Working with Ecollabs Files
Ecollabs files are database files that are structured in one of nine ways—i.e., as one of nine file types. Each type meets a general need; and together the nine types make up a sufficiently complete, fundamental toolset for most team publishing projects. Note that custom file types are also possible—when none of the nine, bundled templates can support a special case.
When a file is loaded, the Craft control panel provides a SAVE and DEL button for temporary, local (device) storage of a working file. However, a working file does not get included in the remote (team) database until the file is "committed" to the local database (and the remote DB is "synced" to the local DB)—using the Final Prep & Publish control panel.
Notice that when no file is loaded, some general-purpose, starter features are accessible below.
Load a template or file of any of the (nine) core types to see (in this screen) more specific helper features related to that file's type. The following list is only an overview of the basic uses of each kind of template.
1-publmgr
Publish Manager: Central collection of all publishing decisions and instructions about a particular, live web doc.
2-srcdoc
Source Document: Edit copy (a plain-text file), intended to be the content of all, or some part, of a live web doc.
3-scrap
Scrap File: A plain-text file used for incidental, non-interactive note-keeping—i.e., for holding details that would not directly get applied to any one, live web doc.
4-event
Calendar Item: Aids both in scheduling future events and in recording past events.
5-prjid
Project Identity: Location for formal statements on a team project's intended purpose.
6-anno
Annotation File: Applies highlights, margin notes, sorting tags and a TOC directly to a publication.
7-memo
Communication Post: Can be used as an in-house memorandum or as a basic unit of an ongoing, team chat.
8-assets
Web-Assets File: Web assets (HTML files, images, fonts, etc.—necessary elements of a live web doc) may be housed within, and organized by, these database files.
9-contact
Contact Data: A storage place for all contact information about one team member or associate.

FILE TYPE: eco-publmgr
Description: This database file type is useful for managing a living publication (live web doc). This type of file can either contain all of the content and code of a resulting web doc, or else combine and organize data from external files—while yet containing some or none of the end content and code.
A loaded publmgr (publish-manager) file will always freshly render something new when the Preview screen is selected. What gets rendered—the publication (web doc)—depends upon the inerplay of content and code provided to it.
While a publmgr file is loaded, the text of an intended publication (including HTML text) may be entered and edited within one or more SOURCE panes of the File Edit screen. When a webpage is rendered, its source code is the auto-combined content of all SOURCE panes in use.
loadconfigs property: Alternatively, the text of a publication may be imported into the SOURCE pane(s) automatically—at the time when a publmgr file is loaded. This can be set up by providing source-document name and location details in the loadconfigs property of the publmgr file—accessed within either the JSO-Pty Edit or the JSON/Text Edit screen. Import-setup procedures are detailed below.
parseconfigs property: When a publication is generated from a publmgr file, the text in the SOURCE pane(s) is pre-render parsed according to instructions provided within the parseconfigs property of the publmgr file. Parse instructions must be composed in JavaScript code and CSS styling. The instructions may be entered directly into the parseconfigs property, or else imported as needed from external (local or remote) ".js" and ".css" web resources. Either way, the final arrangement gets set up within either the JSO-Pty Edit or the JSON/Text Edit screen. Parse-setup procedures are detailed below.
Tip: Pre-render parsing of the content of a publmgr file confers the advantage of reusing publication templates and code in boundless, efficient ways—thereby separating bulk and unreadable matter out from the content of a live publication and allowing free-er expression in the writing and editing of the content.
Pro Tips
  • Template: A custom, publmgr template may be created by prefixing a publmgr file ID with "~TMP1_". Also, if a template is placed in a central, team-organization database, it is then made available from within any project database.
  • DVIZ: A custom, data-visualization file may be created by prefixing a publmgr file ID with "~DVIZ_". Also, if a DVIZ file is placed in a central, team-organization database, it is then made available from within any project database. And, it can generate a visual using the data that it pulls from whichever project database it is being used in.
  • External content mgmt: If a publmgr file is set to import external content when loaded (as described in the import procedures below), the app will perform the import automatically. Subsequently, when a user saves or commits a publmgr file, the app will remove the imported content automatically. However, both of these behaviors can be overridden using the Custom Tools > App control panel.
To import external content into a publmgr SOURCE pane—at the time file is loaded:
  1. Load a publmgr file.
  2. Navigate to Craft > JSO-Pty Edit.
  3. Assign to the commondirpath property of the loadconfigs field a common directory path—i.e., a relative path from the publmgr file to the srcdoc file. If the two files share the same location, a same-location path must be declared. Even so, consider also how the files might get re-organized later—for uses other than in just one database; such as in a computer file system with additional shared resources. Following are three example cases:
    "./" Shared subdirectory
    "../mysubdir2/" Different subdirectory
    "../../mydb2/mysubdir2/" Different database
    A declaration in this field is all that is needed as long as all srcdoc file IDs (filenames) conform to a standard naming convention (see step 4), and as many srcdoc files exist (in one place) as the publmgr file has empty SOURCE panes available to be populated.
  4. Optional customization: The fragsrcxs property of the loadconfigs field accepts an array of multiple values, separated by a comma. Each value in the array corresponds to a SOURCE pane in the publmgr file. An array value helps the app locate the specific srcdoc file that will populate a corresponding SOURCE pane. If no values are found inside the array, the app assumes default values—patterned like the following:
    "-‍s01.txt"
    (as 1st array item)    		 Load into pane SOURCE1 a srcdoc with the same file‑ID root, with a corresponding "-s" suffix number and with a ".txt" filename extension.
    "-‍s02.txt"
    (as 2nd array item)    		 Load into pane SOURCE2 a srcdoc with the same file‑ID root, with a corresponding "-s" suffix number and with a ".txt" filename extension.
    In special cases, srcdoc files might have non-standard file IDs (filenames), might not share uniform paths and might be unnecessary for one or more SOURCE panes of a publmgr file. For these scenarios, assign custom values within the fragsrcxs array—as follows:
    false
    (no quotes)    		 Load nothing into corresponding SOURCE pane.
    null
    (no quotes)    		 Load into corresponding SOURCE pane a conventionally named srcdoc—just as if the value was like one of the two examples of the preceding table.
    "-s03" Load into corresponding SOURCE pane a srcdoc with the same file‑ID root and with a corresponding "-s" suffix number, but with no filename extension.
    ".txt" Load into corresponding SOURCE pane a srcdoc with the same file‑ID root and with a ".txt" filename extension, but with no "-s" suffix number.
    "../../mydb2/mysubdir2/mysrcdoc.txt" Load into corresponding SOURCE pane a srcdoc found at the given, relative filepath—i.e., for determination of srcdoc-file location, disregard publmgr file ID and any value found in the commondirpath property; instead, use only the fragsrcxs value.
  5. Optional publmgr-template imports: The pcfgstemplate and the fragstemplate properties of the loadconfigs field may be used to import pre-designed, JSON data structures. Use the pcfgstemplate property to import parseconfigs data. Use the fragstemplate property to import filefrags data. To either property, assign the local-database filepath of a publmgr file as template—e.g., "/db/file".
To parse SOURCE-pane content of a publmgr file—set to occur prior to rendering:
  1. Load a publmgr file.
  2. Navigate to Craft > JSO-Pty Edit.
  3. No-web-resource option: Using the parseconfigs.scriptsconstr property field controls,
    1. assign to the fncname property an arbitrary name for the intended parse-script function;
    2. select the deftxt option in the associated, second select list;
    3. use the same field now to compose a pre-render parse script in JavaScript code.
      Note: The function accepts as its first argument the content string of a SOURCE pane—and must return the result of any transformations made to that string.
    Web-resource-dependent option: Using the parseconfigs.scriptsconstr property field controls,
    1. assign to the fncname property an arbitrary name for the intended parse-script function;
    2. assign to the filepath property a relative or absolute URL to a JavaScript-file (module) resource.
      Note: The JavaScript file must constitute a function that accepts as its first argument the content string of a SOURCE pane—and that returns the result of any transformations made to that string.
    Note: SOURCE-pane content gets parsed during the rendering of the webpage. But, in this option, any external parse-scripts being used are retrieved and preloaded when the publmgr file is loaded. Thus, if any script-file references are changed or added when editing, the working file must then be saved again (reloaded) for the parsers to have an effect.
  4. Using the parseconfigs.scriptsincl property field controls,
    1. assign to the fncname property the same name given to the new, parse-script function;
    2. assign to the applytofrag property an array of boolean values—each value corresponding to a SOURCE pane, and each value either activating (value: true) or deactivating (value: false) the parse-script function's effect on the content of that SOURCE pane.
  5. As both the scriptsconstr and the scriptsincl property types are arrays of objects, additional parse-script functions may be added and configured using the associated, first select list of each field.
To inject styling info into publmgr-file output—set to occur prior to rendering, but after any scripted, pre-render parsing:
  1. Load a publmgr file.
  2. Navigate to Craft > JSO-Pty Edit.
  3. No-web-resource option:
    Using the parseconfigs.linksconstr property field controls,
    1. assign to the insertposition property a value that indicates where the styling info is to be inserted—as follows:
      0  or false Do not inject styling info.
      1  or "top" Insert at the beginning of the resulting output.
      2  or "head" Append to the first <head> element, if found, in the resulting output—otherwise, insert at the beginning.
      3  or "body" Append to the first <body> element, if found, in the resulting output—otherwise, insert at the end.
      4  or "bottom" Insert at the end of the resulting output.
    2. select the htmllinkinserts option in the associated select list;
    3. use the field now to compose the CSS styling that will be injected.
    Web-resource-dependent option:
    Using the parseconfigs.linksconstr property field controls, assign to the linksinclrender property the boolean value true.
    Using the parseconfigs.linksincl property field controls,
    1. assign to the filepath property the relative or absolute URL of a CSS-file resource;
    2. assign to the applytorender property a boolean value—true, to activate the insertion of the style info; false, to deactivate it.
    As the linksincl property type is an array of objects, additional CSS-file links may be added and configured using the associated select list of this field.
    Note: This option generates links to CSS-file resources and applies their styling to the rendered webpage without injecting any style data into the publmgr-file output. This option may be applied either with or without also injecting style data via the no-web-resource option.
  4. Optional @import expansion: When providing styling info in the htmllinkinserts property of the parseconfigs.linksconstr field (a-la the no-web-resource option of step 3), if one or more @import at-rules are included, each @import line can be expanded into (replaced with) the style content found at the @import-URL resource, if available.
    How? Enable this by assigning to the atimportexpand property of the loadconfigs field the boolean value true.
    DATA RETRIEVAL OPTIONS
    Expansion occurs during style-data injection (prior to rendering). But, retrieval of the style data occurs either at the time when a publmgr file is loaded, or else during a style-preloading phase—depending on the retrieval method, which is configured as follows:
    1. If simply using DB queries or HTTP requests to generate the style data (at the time file is loaded), the publmgr file must be loaded with the linksinclrender property (of the parseconfigs.linksconstr field) set to false.
      Tip: This behavior can be overridden using the Custom Tools > App control panel.
    2. Otherwise, if generating style data from preloaded CSSOM styleSheets, CSS-file resources must be specified using the parseconfigs.linksincl field (a-la the web-resource-dependent option of step 3)—and they must be preloaded and their style data captured, then injected. This is accomplished in two passes: by first rendering the webpage in Preview with linksinclrender activated, then rendering the webpage in Preview a second time with linksinclrender de-activated.
FILE TYPE: eco-srcdoc
Description: This database file type is useful as a source of content for one or more publications (web docs).
Presuming that a publmgr file has been set up to import and parse external data for a publication (web doc), a srcdoc (source-document) file provides that external data. It can contain nothing more than pure content—free of markup, styling or anything else that distracts from the composition process. Also, strategic use of srcdoc files allows for the organizing of content or code into logical chunks—in order to accomodate a division of tasks based on team dynamics or other workflow needs.
If the file ID of a srcdoc has a dashed s-digit-digit ("-s01", "-s02", etc.) on the end of it, it will be recognized automatically by a publmgr file having the same file‑ID root. The number indicates to which publmgr SOURCE pane the srcdoc-file's content will be imported—e.g., "mysrcdoc-s01.txt" to SOURCE1, "mysrcdoc-s02.txt" to SOURCE2, etc.
Pro Tip
  • Template: A custom, srcdoc template may be created by prefixing a srcdoc file ID with "~TMP2_". Also, if a template is placed in a central, team-organization database, it is then made available from within any project database.
FILE TYPE: eco-scrap
Description: This database file type serves as a ready-use notepad—used in a way similar to a common, legal pad. Thus, if no legal pad is handy when needing to take down some notes, a scrap file will substitute nicely.
A scrap file is lightweight—essentially holding just plain text. It does not have any inherent relationship to the project files of the database in which it resides and, so, functions well as a place apart from any one project for safe-keeping random, miscellaneous thoughts.
On the other hand, because a scrap file cannot be tied to a project, any project-critical data—data that is meant, eventually, to have some effect on a publication—that gets placed in one will need to be moved again to a file type that can be formally tied to the project. With some pre-planning, this added step can be avoided.
Pro Tip
  • Template: A custom, scrap template may be created by prefixing a scrap file ID with "~TMP3_". Also, if a template is placed in a central, team-organization database, it is then made available from within any project database.
FILE TYPE: eco-event
Description: This database file type is useful for scheduling future events and/or recording current and past events. Multiple event files may be used to generate various kinds of calendars—e.g., as a sequential list, in a familiar month-by-month layout, etc.
Advantages: An event file can be adapted for other, non-calendaring purposes—such as writing spreadsheet-like data rows (entries) in real time, or compiling research notes over an extended time period.
Pro Tips
  • Template: A custom, event template may be created by prefixing an event file ID with "~TMP4_". Also, if a template is placed in a central, team-organization database, it is then made available from within any project database.
  • event file ID: The file ID of a new event file is automatically generated by the app. The string conforms to a special, user-time-code pattern for efficient sorting—e.g., "~E20200315T083045abf".
FILE TYPE: eco-prjid
Description: This database file type is useful for defining a project's purpose. When a project idea is first conceived by someone on the team, the purpose of the potential, new project—shaped into words—can be entered into the fields of a prjid (project identity) file.
Additionally, a prjid file "announces" the contents of a database. When an existing database is selected by an interested visitor at the web app entryway, a preview block is displayed. For each project (or, team-organization) database, the preview block can be customized by creating a corresponding prjid file—one that resides, either, within the same database, or else with others in a central, team-organization database.
Note that the NAME field of a prjid file can accept any character and any string combination—including a blank string. However, to ensure integration of this file type within the app, the file ID must have a "~DBID_" prefix, and the root must be identical to the name of the database that it describes. Thus, the constraints for a file‑ID root are the same constraints imposed upon a database name—i.e., it must begin with a letter and comprise only lowercase alphanumerics, an underscore (_) and a dash (-).
Pro Tip

FILE TYPE: eco-anno
Description: This database file type is useful for applying to a digital publication: sorting tags, a Table of Contents, a chapter-paragraph counter, and colorful, visual marks—i.e., draft-editing marks such as highlights, strikes and margin notes.
An anno (a static, annotation) file takes markdown data (minimal, plain-text markup characters) that gets converted into HTML, inline-markup tags. When the data is activated by a user, the markup tags are applied to a publication, and the publication webpage is re-rendered in the Preview screen—making the intended highlights, strikes and margin notes visible.
While that sounds unduly technical, the basic function of an anno file is simple: Any word, phrase or line of text in a publication that gets copy-pasted into the HIGHLIGHTS field of this file will get highlighted in the publication. Extended features are further detailed in Pro Tips below.
In practice, this file type helps to make editor's notes on a project creation more visual and effective. Followed routinely, this practice also accumulates space-efficient, easy-access records of a developing publication's evolutionary stages.
Pro Tips
  • Template: A custom, anno template may be created by prefixing an anno file ID with "~TMP6_". Also, if a template is placed in a central, team-organization database, it is then made available from within any project database.
  • anno file ID: The file ID of a new anno file is automatically generated by the app. The string conforms to a special, user-time-code pattern for efficient sorting—e.g., "~a20200315T083045abf".
  • LINK TO field: Enter a filename here—specifically, the filename root of a web doc (residing anywhere in the same database) that, when previewed, is to be decorated with the annotations detailed in the anno file. To specify a web-doc version to be annotated, first give the web-doc filename a semantic version suffix; then, include the same suffix in this field—e.g., "mypubl03-v0.1.5". Otherwise, if the filename in this field is the root only (with no suffix), all existing versions with the same filename root will be affected.
  • TAGS field: Tags are useful for organizing, sorting and running searches on database files in non-standard ways. List any number of tags here, separated by a space or comma. Tag names can be arbitrary but must contain only alphanumeric characters.
  • TOC FMT field: Use this field to generate and format a Table of Contents (TOC), chapter numbering and paragraph counter within a book-like web doc (ebook). A few, general-purpose configurations are provided in the data list—when the input field is focused and empty. Create a more custom configuration by manually entering a character pattern based on one of the provided options.
  • STRIKES, INSERTS & HIGHLIGHTS field: Each line of text in this field is read as a search string (exceptions noted below)—for locating a string of text to be annotated within the web doc. Along with each search string, optionally specify the desired type of annotation (HTML, inline-markup tags) to be applied—directly after, and on the same line as, the search string. Declare the annotation type (tag name) using markdown markers wrapped in curly braces—as follows:
    {=} or {::} <span>...</span> for custom markup
    {==} or (blank) <mark>...</mark> for highlighting
    {**} or {__} <strong>...</strong> for making bold
    {*} or {_} <em>...</em> for making italic
    {~~} <s>...</s> for striking out
    {++} <ins>...</ins> for inserting new text—i.e., a "margin note"
    • anno insert: Placement of an insert marker is one exception to the rule. Do not place an insert marker on the same line as a search string like for the other markers; rather, place it directly after a line of new text—i.e., at the end of the "margin note" that is to be inserted. Thus, a complete insert annotation requires two lines—one line for the search string (which may or may not also declare another annotation, like a strike), and a second line for the insert text.
    • anno switch: An annotation marker (wrapped in curly braces) may also be placed on its own line—apart from any one, single search string. Doing this will affect all the lines of search strings that follow and that do not have their own, specific annotation marker.
    • anno attribute: Within curly braces, specify a desired doc-element ID, classname or background color to be applied along with an (inline-markup) annotation—right after any markdown marker, if present—as follows:
      {Beige} anno switch—placed on its own line
      {==Salmon} a highlighting (<mark>) tag is given a salmon color
      {*\#ff33cc} the escaped hash (\#) indicates an HTML/CSS color code
      {__#mainpoint} the unescaped hash (#) indicates an ID
      {++.marginnote} the dot (.) indicates a classname
    • Phrase targeting with RegExp: For finer search control, enter a JS Regular Expression (RegExp) search pattern (e.g., "/^canary|^finch/gim") on any line by itself—i.e., without a curly-brace marker. In order to couple the search with a custom annotation type, enter an anno switch on the line preceding the RegExp search pattern. And, if that switch is not to be applied to any subsequent searches, enter another, "reset" anno switch on the line following the RegExp search pattern.
       When searching for a short phrase or single word that can occur more than once in a document, target the precise occurrence by using a strategic RegExp search pattern. For example, make a RegExp for the one phrase that has a unique, trailing string—like, "/friend(?= of my sister's hairdresser)/". In this example, the characters within the parentheses and after the "?=" are unaffected words that immediately follow the targeted word "friend".
    • Paragraph targeting with "\v": Strategic use of a RegExp search pattern may not be necessary in many cases. Another targeting option is searching for multiple, disconnected phrases within a single paragraph by making use of the ascii, vertical-tab character, "\v".
       Enter each of two or more disconnected phrases on its own line; list them in the same order that they occur in the paragraph. Then, enter a "\v" at the end of every phrase except the last one. Optionally, enter a curly-brace marker at the end of the last phrase to specify a highlight color or other inline markup/style for the whole block of phrases.
       The "\v" at the end of a string indicates to the parser that the string on the next line is found within the same paragraph—following the previous string yet separated from that string by unaffected, inline characters.
    • Comment: Insert a comment (e.g., a note to self) anywhere throughout by prepending the comment text with two forward slashes—e.g., "//comment text". Commented text has no effect on the publication being annotated. Note that this feature is useful also for experimentally deactivating a line with a search string, a search pattern, an insert or a switch.
FILE TYPE: eco-memo
Description: This database file type is useful for "private" in-house communications and for public feedback on a team's live publication(s).
A (static, communication) memo file is created and added to the database by the app when a team member or visitor posts a comment on the Discuss screen for a (loaded) publication.
Additionally, a user may use a memo file to send a message with a more specific purpose—such as sharing a requested report with the team leader, handing out assignments to individual team members, or pitching a not-yet-existing project idea to a select few members of the team.
Caution: All database files are accessible to all team members—and possibly (if the database is ever made public) to anyone. Therefore, no memo-file communication should be considered private between just the sender and the recipient(s) named in the memo.
Advantages: Note that, while ordinary e‑mail or texting can be a more efficient means of communication in many instances, using memo files yields other benefits. All memo files of a project are kept together, within the project database; so, maintaining an easy-access record of all project-related communications is convenient. Also, a memo file can be adapted for other, non-communication purposes—such as writing spreadsheet-like data rows (entries) in real time, or compiling research notes over an extended time period.
